# Thilenius Personal Infrastructure

Welcome to the `Stephen` workspace. This monorepo ecosystem comprises five tightly integrated projects that form a robust, full-stack smart-home dashboard, remote access portal, and hardware IoT architecture. 

Below is an overview of the projects, their distinct roles, and a detailed map of how they weave together to form the complete infrastructure.

## The Five Projects

### 1. `stephenBackEnd` (The Central Hub)
A Node.js/Express server (TypeScript) serving as the primary data broker and logic engine. It interfaces with the database (PocketBase), manages job scheduling (cron/suncalc), handles Modbus/Serial protocols, and provides a unified WebSocket and REST API for all clients and connected hardware.

### 2. `stephenFrontEnd` (The Web Portal)
A comprehensive React 18/Vite web application. It acts as the "birds-eye view" of the entire system for the end-user, rendering complex interactive charts, photo galleries, and real-time device dashboards. It uses `stephenBackEnd` to stream all its live and historical data.

### 3. `4inDisplay_S3` (The Hardware Nodes)
A massive PlatformIO (C++) firmware codebase for ESP32-S3 controllers equipped with 4-inch TFT touchscreens. This *single* codebase is compiled into entirely different firmwares based on preprocessor environment macros (e.g., `COFFEE`, `SPRINKLER`, `DESK`, `POWERMETER`), depending on the hardware it is flashed to.

### 4. `espserver` (The Embedded Configuration UI)
A lightweight React Single Page App (SPA). Unlike the main Frontend, `espserver` is compiled as a static build and flashed *directly* onto the ESP32's LittleFS internal filesystem. It allows users operating on the local network space to access the embedded hardware configuration directly without relying on external servers.

### 5. `sprinklerApp` (The Mobile Controller)
A React Native mobile application custom-built specifically for configuring, scheduling, and manually overriding the physical remote sprinkler hardware connected to the infrastructure.

## 🕸️ How They Interact

The true power of this workspace is how the systems connect. The architecture utilizes a "hub-and-spoke" model with the Backend at the center, supplemented by localized peer-to-peer and embedded hardware interactions.

### 📡 Client-to-Server Interactions
- **UI Applications (`stephenFrontEnd` & `sprinklerApp`) ↔ `stephenBackEnd`:** 
  The frontend and mobile apps maintain persistent WebSocket connections with the Backend. The web portal fetches historical states via REST, while WebSockets push sub-second state changes instantly (e.g., a power meter spikes, and the Backend pushes it live to the web charts). The mobile app interacts with the backend to configure complex database scheduling for the sprinkler hardware context.
- **Backend ↔ Database:** 
  All identity management, long-term telemetry storage, and heavy media serving are negotiated exclusively by the Backend against the centralized PocketBase instance.

### 🔌 Firmware-to-Backend Interactions 
The `4inDisplay_S3` firmware connects aggressively to `stephenBackEnd` via WebSockets and HTTP routes to synchronize hardware states. However, the interactions differ drastically depending on the chosen PlatformIO build configuration:
- **`DESK` / `COFFEE` Builds (Interactive Hubs):** 
  These act as remote smart hubs. They subscribe to the Backend to receive the *entire* house state and display it across a custom TFT Tab UI. They also publish commands upward (e.g., tapping "Turn On Coffee" on the desk display tells the Backend to relay a command to the ESP32 hooked to the coffee pot).
- **`POWERMETER` Build (Aggressive Sensor Node):** 
  Acts as a high-frequency sensor unit. It primarily focuses on streaming dense, continuous data *up* to the Backend (voltage, current, wattage), leaving the heavy-lifting of data aggregation and storage to the server so the `stephenFrontEnd` can easily pull down cleaned data later for charting.
- **`SPRINKLER` Build (Execution Node):** 
  A hardware execution unit that prioritizes listening. It waits for the Backend to dictate scheduling lifecycle triggers or manual overrides originating from the `sprinklerApp` or `stephenFrontEnd` and clicks over physical relays. 

### 🎛️ The Local Hardware Escape Hatch (Server Bypass)
- **`4inDisplay_S3` ↔ `espserver`:** 
  If the Backend is unreachable or Internet connectivity drops, the system employs localized hosting. When you navigate to a specific ESP32 node's local IP address, the `ESPAsyncWebServer` running in the C++ firmware loads the static React assets generated by `espserver` straight from its internal SPIFFS/LittleFS memory.
  Once loaded into your browser, `espserver` opens a localized WebSocket *directly against the ESP32 hardware*, querying its WiFi variables, file system status, or hardware diagnostic readouts. This acts as an autonomous provisioning tool and localized escape hatch.

### 🌊 Overarching Data Flow Example
1. You open the **Mobile App** (`sprinklerApp`) on your phone and tap "Water Zone 1".
2. The app issues an authenticated REST command to **The Hub** (`stephenBackEnd`).
3. The Backend updates the persistent database to record the event and broadcasts a "Zone 1 Active" state change over WebSockets to all subscribed network clients.
4. The physical **ESP32 Controller** (`4inDisplay_S3` with `-D SPRINKLER` macro enabled) receives the event, physically flips its GPIO relay to start the water, and echoes a hardware confirmation back to the Backend.
5. The **Web Portal** (`stephenFrontEnd`) and an indoor display panel (`4inDisplay_S3` with `-D DESK`) receive the broadcast from the Backend and update their UIs to indicate that water is currently running outside.
6. A maintenance technician onsite can input the Sprinkler's IP address directly into a tablet to load the internal **Embedded Configuration UI** (`espserver`), testing the physical relay pins manually without passing a single packet through the broader Internet or centralized server.
